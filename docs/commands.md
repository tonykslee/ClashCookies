# Commands Reference

- `/help [command:<name>] [visibility:private|public]` - List command docs/examples. Default visibility is private.
- `/ban add player:<playerTag> user:<discordUser> clan:<trackedClan> [reason:<text>] [duration:<3mo|2w|10d|12h>]`, `/ban list`, and `/ban remove player:<playerTag> user:<discordUser>` - Maintain persisted guild-scoped ban records for either one player tag or one Discord user at a time. `player` and `user` are mutually exclusive and runtime validation rejects requests that provide neither or both. Player tags are normalized with the same Clash-tag helper used by link/accounts commands. `clan` is optional tracked-clan context, autocompletes from tracked clans, and is displayed in `/ban list` plus banned-player join alerts; it does not change which player tags or Discord users are banned. Ban action logs route through `/bot-logs type:ban-log`, while banned-player join alerts route through `/bot-logs type:ban-join-alert`. `duration` is optional; omit it for an indefinite ban, and invalid duration strings are rejected with a clear error. `/ban list` shows active bans only, including optional clan context and linked player tags for user bans, and paginates safely under Discord embed limits. Removing a user ban does not copy linked player tags into the ban row; linked tags are resolved from `PlayerLink` when rendering and checking.
- `/emoji [name:<emoji_name>] [react:<message-id>] [emoji:<emoji-token-or-url>] [short-code:<name>]` - Resolve one bot application emoji by name, react to a message in the current channel when `react` is provided, add a new bot application emoji when `emoji` + `short-code` are provided (admin-only by default), or browse a paginated list of all bot application emojis when `name` is omitted. React successes confirm privately to the invoker and are logged to `/bot-logs` when configured. No-arg list pages are capped to ~2500 characters of emoji-entry content and rendered in compact row-wise 3 columns sorted by shortcode length. For name-only resolves, `visibility:public` returns only the rendered emoji token while private keeps the detailed shortcode/raw-token output.
- `/telemetry report [period:24h|7d|30d] [timezone:<ianaTz>]` - Show aggregate telemetry report (usage, latency, failures, API health) for the selected window.
- `/telemetry schedule set target-channel:<channel> cadence-hours:<n> [timezone:<ianaTz>] [enabled:true|false]` - Configure scheduled telemetry report posting.
- `/telemetry schedule show` - Show current telemetry schedule config and last posted window.
- `/telemetry schedule disable` - Disable scheduled telemetry report posting.
- `/telemetry schedule run-now` - Post one report for the current completed schedule window (idempotent per window).
- `/permission add command:<name> role:<discordRole> [role2] [role3] [role4] [role5]` - Allow one or more roles to use a command.
- `/permission remove command:<name> role:<discordRole>` - Remove a role from a command whitelist.
- `/permission list [command:<name>]` - List role policy for one command target, or all if omitted.
- `/lastseen tag:<playerTag>` - Show a player's last seen activity, with drill-down button for tracked signal timestamps.
- `/inactive days:<number> [consecutive:true|false] [in-clan:true|false] [clan:<tag>]` - List players inactive for N days from DB-backed current membership plus persisted activity/last-seen rows. `in-clan:true` is the default and only shows players still in the tracked clan; `in-clan:false` only shows players who have left when the current-membership table can confirm that absence. `consecutive:true` narrows to continuous inactivity across the requested window. Paginated replies include a button that toggles between player tags and compact weights.
- `/inactive wars:<number> [consecutive:true|false] [in-clan:true|false] [clan:<tag>]` - List tracked-clan members who missed both attacks in at least one of the last N ended tracked wars (requires war-history tracking window). `in-clan:true` is the default and only shows current members; `in-clan:false` only shows players who are no longer in the tracked clan when current-membership data can confirm that absence. The clan filter scopes days mode, wars mode, and the combined days+wars mode. Wars output groups by clan, then missed-count, and shows the linked Discord user, player tag, and missed-war emoji sequence. Paginated replies include a button that toggles between player tags and compact weights. When no rows match, the bot includes a short diagnostic note showing whether ended-war rows and participation rows were found.
- `/clan-health [visibility:private|public] tag:<trackedClanTag>` - Leadership snapshot from persisted data only: last-30 ended-war match metrics with FWA win/loss, BL, and MM breakdown plus a BL-inclusive match rate, inactive counts (missed-both in at least one of the last 3 ended FWA wars + last-seen >=6d), and missing Discord links among observed members.
- `/repwork since:<duration> [visibility:private|public]` - Report FWA leader rep-work activity over a recent window using persisted attribution snapshots only. `duration` accepts positive integers with `d`, `w`, or `mo` suffixes such as `7d`, `4w`, or `2mo`; `mo` is treated as 30 days and invalid input returns `Use a duration like 7d, 4w, or 2mo.`. Bases and mail timing use the prep-time-left value captured when the activity was attributed, with Discord mails counting checklist reactions and in-game mails counting successful `/fwa match` mail sends. Sync stats show `syncs participated` and `clan claims` from persisted sync claim rows. The report also shows each user's top 3 bot commands in the window and omits users who only have command usage with no rep-work or sync activity.
- `/role-users role:<discordRole>` - List users in a role with pagination.
- `/layout [th:<number>] [type:RISINGDAWN|BASIC|ICE] [edit:<layout-link>] [img-url:<url>]` - Fetch or list stored FWA layouts; `edit` upserts are admin-only at runtime, and `img-url` is only valid when `edit` is provided.
- `/dump [edit:<link>] [slot:<1|2|3>]` - Show one or more short clan summaries plus the stored guild dump links as plain text wrapped in angle brackets to suppress Discord embeds. Each configured link is rendered in slot order and separated by `------------`. `edit` is admin-only at runtime and stores one link per guild/server in slot `1`, `2`, or `3`; `/dump` falls back to cached clan summary data when live clan fetch is unavailable. Use `/dump slot:X` as an admin to delete a specific slot, or `/dump edit:<link> slot:X` to replace a specific slot.
- `/clan configure tag:<tag> [lose-style:triple-top-30|traditional] [mail-channel:<discordChannel>] [log-channel:<discordChannel>] [leader-channel:<discordChannel>] [clan-role:<discordRole>] [lead-role:<discordRole>] [clan-badge:<emoji>] [short-name:<abbr>] [reps:<tag-list>]` - Add/update tracked clan settings.
- `/clan cwl-tags cwl-tags:[<tag-array-or-comma-list>]` - Add one batch of seasonal CWL tracked clans (supports tags with or without `#`), with partial-success summary buckets for added/already-existing/invalid/duplicates.
- `/clan raid-tags raid-tags:[<tag-array-or-comma-list>] [upgrades:<number>]` - Add or update one batch of RAIDS tracked clans. Single-tag writes can optionally persist manual upgrades, and write-time clan profile lookups best-effort cache join status for the RAIDS registry.
- `/clan remove tag:<tag> [type:FWA|CWL|RAIDS]` - Remove tracked clan from FWA, current-season CWL, or RAIDS registry. When `type` is omitted and a tag exists in multiple registries, command returns an explicit ambiguity prompt instead of deleting.
- `/clan list [type:FWA|CWL|RAIDS] [display:minimal|detailed]` - List tracked clans and settings. Without `type`, shows one embed grouped by FWA/CWL/RAIDS with clickable clan titles, persisted current member counts when available, and a member-count `Refresh` button; `display` is ignored unless `type` is set. `type:FWA|CWL|RAIDS` defaults to the minimal overview-style section. `display:minimal` shows the overview-style section with member counts, no `leadRole`, and the member-count `Refresh` button. CWL minimal rows use the CWL league emoji as the leading marker when ranked, keep `-` for unranked/unknown, and show spin status as emoji only. `display:detailed` opens the drilldown: detailed FWA still shows `leadRole` and configured reps, detailed CWL now shows the clan link/tag, CWL league emoji, spin status emoji, observed CWL roster count, current clan member count, and active roster link/title with a manual refresh button, and detailed RAIDS keeps its existing metadata refresh button behavior. Detailed CWL rows sort by CWL league descending, then active roster title ascending.
- `/raids overview [type:raids|fwa|custom] [clan:<clanTag>]` - Show the tracked RAIDS dashboard focused on raid status. The overview includes a tracked-clan dropdown, refresh button, clickable clan titles, and compact per-clan summary lines. Clan titles can show a shield metric like `🛡️46` for the highest attacks used against us in one defending raid. The intel line no longer starts with `⚔️`, and it is hidden once a clan has completed at least one full offensive raid. A compact offense line shows total attacks, destroyed districts out of 9, and average attacks per completed raid. Medal estimates show Offensive from cleared districts/capital peaks and attacks used; Defense appears only when the raid payload includes `defensiveReward`, and Total appears only when both values are available. `⚔️` means a raid has started and is incomplete, `🌄` means at least one raid is completed and no raid is currently ongoing, and no emoji means no raid started/completed. `type:raids` uses raid-tracked clans and raid autocomplete, `type:fwa` uses tracked FWA clans and FWA autocomplete, and `type:custom` accepts any valid clan tag from the `clan` option without requiring tracked membership. Rows sort by ongoing raid, raids completed, current-season default layout count, and then the tracked display order. Selecting a clan switches to a single-clan view with the same refresh button plus a back-to-overview button; when the live raid-season source can resolve it, the single-clan view also shows attack-log and defense-log detail sections, and open attacking clans include a requirements sub-line when their live clan metadata is available. `/raids overview clan:<tag>` keeps the detailed attack/completed stats, and the drilldown adds a compact grade-grouped intel summary above the attack/defense sections when marks exist, and the defending section shows attacks used plus districts remaining for each attacking clan with `🔓` for open clans and `🔒` for closed, invite-only, or unknown clans. `/raids intel [clan:<trackedRaidClanTag>] [upgrades:<1000-3331>] [capital_peak:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [barbarian_camp:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [wizard_valley:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [balloon_lagoon:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [builders_workshop:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [dragon_cliffs:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [golem_quarry:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [skeleton_park:<Default|Custom - Hard|Custom - Medium|Custom - Easy>] [goblin_mines:<Default|Custom - Hard|Custom - Medium|Custom - Easy>]` opens a raid-intel view for the selected tracked clan with attack-log defender sections, district hall levels when available, saved layout grades, optional pre-marked district args, and interactive district controls. The tracked-clan line stays focused on the source clan, each defender section header renders its saved defender upgrades inline as `| 🏘️ {upgrades or —}`, and supplying `upgrades:<number>` saves that value to the single resolved defender clan when there is exactly one; if the attacked clan is ambiguous, the bot leaves persisted marks unchanged and tells you it could not save the upgrades.
- `/raids roster add tag:<playerTag(s)>` - Add one or more normalized player tags to the guild's RAIDS roster. The tag input accepts comma-separated, space-separated, or mixed-separated tags and reports added, already on roster, and invalid tags.
- `/raids roster status` - Show persisted RAIDS roster members with Town Hall, linked Discord user, and current raid completion from shared todo snapshot data when available. Large rosters are chunked across multiple embeds to stay within Discord limits.
- `/sheet link sheet_id_or_url:<id-or-url> [tab:<tab-name>] [mode:actual|war]` - Link or relink sheet; mode is optional.
- `/sheet show [mode:actual|war]` - Show linked sheet settings (single mode or all).
- `/sheet unlink [mode:actual|war]` - Remove one mode link or all links.
- `/sheet refresh mode:actual|war` - Trigger mode-specific Apps Script raw feed refresh. Disabled when `POLLING_MODE=mirror`.
- `/compo advice clan:<tracked-clan> [mode:actual|war]` - Pull advice using mode-specific sheet link. `mode:actual` uses the same persisted ACTUAL current-member weight resolution as `/compo state`, so feed misses can still fall back through `FwaPlayerCatalog` and `PlayerCurrent` before the existing deferred/war ACTUAL fallbacks. ACTUAL advice defaults to `Auto-Detect Band`; `Raw Data`, `Auto-Detect Band`, `Best Fit`, and `Custom` remain available with inline buttons. In ACTUAL `auto`/`best` views, the embed labels both the resolved roster weight and the projected 50-player scoring weight so it is clear when recommendation scoring is using a projected roster.
- `/compo state [mode:actual|war]` - Render AllianceDashboard state as an attached PNG image. `mode:war` reads persisted feed-backed tracked-clan roster state from `FwaTrackedClanWarRosterCurrent` + `FwaTrackedClanWarRosterMemberCurrent` + `HeatMapRef` and its inline `Refresh Data` button refreshes tracked-clan war-roster feed state only before rerendering from DB. `mode:actual` now reads persisted ACTUAL current-member state from `TrackedClan` + `FwaClanMemberCurrent` + `FwaPlayerCatalog` + `PlayerCurrent` + `HeatMapRef`, uses the higher of FWA Stats and deferred/manual weight before falling back to lower-confidence sources such as tracked WAR effective weight and missing, still renders when some weights are missing, and its inline `Refresh Data` button refreshes tracked-clan ACTUAL member/weight state plus live CoC member counts for all tracked clans before rerendering from DB. ACTUAL state defaults to `Auto-Detect Band`, and the inline buttons now only switch between `Raw Data` and `Auto-Detect Band`. Raw Data shows current resolved composition counts instead of HeatMapRef deltas.
- `/compo heatmapref [mode:fwa|blacklist]` - Render the persisted HeatMapRef table as an attached PNG image with `TH11+`, `Match%`, and `Clans` columns, and use the inline `Copy Table` button to get a formatted CSV text block with `WeightMin`, `WeightMax`, `TH11+`, `Match%`, and `# Clans` without triggering any rebuild/update work. `mode:blacklist` switches to the persisted blacklist profile rows, which add sample counts, source/opponent counts, confidence, and generated timestamps; if no blacklist rows exist yet, the command points admins to `/fwa blacklist-profile rebuild`.
- `/compo place weight:<value>` - Suggest placement options from ACTUAL composition data. Accepts formats like `145000`, `145,000`, or `145k`, maps the input into the same stable display buckets used by `/compo`, and reads persisted ACTUAL current-member state from `TrackedClan` + `FwaClanMemberCurrent` + `HeatMapRef` using the same ACTUAL Auto-Detect projection path as `/compo state` and `/compo advice`, so projected totals select the target band while resolved-roster deltas drive recommendations. The main result still shows `Recommended`, `Vacancy`, and `Composition`, and may include a compact `Possible replacements` summary per clan when replacement candidates exist. Every `/compo place` response includes an inline `Refresh Data` button that explicitly refreshes tracked-clan ACTUAL member/weight state plus live CoC member counts for all tracked clans before rerendering from DB. A `Show replacements` button opens an ephemeral, paginated player-level drill-down from the same stored data. Replacement reasons are only `🧍 filler`, `😴 inactive`, and `📵 unlinked`; stale or missing weight are not replacement reasons. Linked drill-down rows show the Discord mention first, while unlinked rows show no mention and include `📵 unlinked`. `/compo state mode:war` remains DB-backed separately.
- `/compo fill` - Recommend filler placements using the existing planner, persisted ACTUAL tracked-clan state, and guild filler accounts only. The render groups recommended moves by destination clan and separately shows remaining open slots, unused available fillers, unavailable fillers with reason codes, and excluded/missing-weight fillers.
- `/cc player tag:<tag>` - Build `https://cc.fwafarm.com/cc_n/member.php?tag=<tag>`.
- `/cc clan tag:<tag>` - Build `https://cc.fwafarm.com/cc_n/clan.php?tag=<tag>`.
- `/notify war clan-tag:<tag> target-channel:<channel> [role:<discordRole>]` - Enable war-state event logs (war start, battle day, war end) for a clan in a selected channel. Optional role is pinged when event logs are posted.
- `/notify war-preview clan-tag:<tag> event:<war_started|battle_day|war_ended> [source:current|last]` - Show an ephemeral preview embed and confirm before posting publicly to the configured notify channel.
- `/warplan set clan-tag:<tag> match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>` - Opens a modal to edit one clan custom plan. `clan-tag` supports autocomplete from tracked clans. Modal supports normal new lines, markdown (bold/italic/code/code blocks), emoji shortcodes, direct image/GIF links, plus optional compliance fields: `minimum clan stars before tripling non-mirror` and `all bases open for 3 star time-left` (`H` or `Hh`, range `0..24`). Those two compliance inputs are prefilled from the effective config for the selected match set (custom -> editable default -> built-in fallback defaults).
- `/warplan show [clan-tag:<tag>] [match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>]` - With `clan-tag`, show effective clan plans (clan custom -> editable guild default -> built-in fallback) including resolved compliance gate values (`nonMirrorTripleMinClanStars`, `allBasesOpenHoursLeft`). Without `clan-tag`, shows a paginated guild-scoped tracked-clan overview listing which exact custom override types exist (`FWA-WIN`, `FWA-LOSS-TRIPLE_TOP_30`, `FWA-LOSS-TRADITIONAL`, `BL`, `MM`) or that defaults are used for all match types.
- `/warplan reset clan-tag:<tag> [match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>]` - Reset clan custom plans. With no match-type, resets all custom plans for that clan.
- `/warplan set-default match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>` - Opens a modal to edit one guild default plan used when a clan has no custom plan, with the same optional compliance fields and effective-config prefill behavior as `/warplan set`.
- `/warplan show-default [match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>]` - Show editable guild defaults and effective fallbacks, including resolved compliance gate defaults when unset.
- `/warplan reset-default [match-type:<BL|MM|FWA|FWA_WIN|FWA_LOSE_TRIPLE_TOP_30|FWA_LOSE_TRADITIONAL>]` - Reset editable guild defaults back to built-in fallback text.
- `/war history clan-tag:<tag> [limit:<number>]` - Show recent clan-level war history from stored war records.
- `/war war-id clan-tag:<tag> war-id:<number>` - Export stored war lookup payload for one clan-scoped war ID as CSV. `war-id` supports autocomplete after `clan-tag` is selected and returns recent ended wars for that clan only (up to 10).
- `/cwl members clan:<trackedCwlClanTag> [inwar:true]` - Show the persisted observed current-season CWL roster for one tracked CWL clan, optionally narrowed to the current/prep lineup only, plus the linked signup roster context when available. The output includes the observed CWL member count, the signed-up + spun count, and a copy/paste exclusion string for CWL players that are included but not signed up.
- `/autorole refresh [user:<discordUser>] [role:<discordRole>]`, `/autorole config show`, `/autorole config set ...`, `/autorole rules list|add|edit|remove`, and `/autorole exclusions list|add-user|remove-user|add-role|remove-role` - `/autorole refresh` defaults to FWA Leader role + Administrator and performs a tracked FWA clan membership refresh when run with no args; `user` and `role` scoped refreshes stay admin-only, and both scope arguments together are rejected. `/autorole config`, `/autorole rules`, and `/autorole exclusions` stay admin-only by default. `/autorole config set` can also set or clear the generic CWL clan role used for users with eligible linked accounts in a tracked current-season CWL clan, configure a visitor/non-member role with `non-member-role` and `non-member-enabled`, plus a stale CLAN-role removal delay in minutes. Nickname sync is enabled with `apply_nicknames:true` plus a `nickname_template`, for example `{player} | {trackedClans}`.
- `/roster create category:<CWL|FWA> clan:<trackedClanTag> [name:<text>] [title:<text>] [timezone:<ianaTz>] [start_time:YYYY-MM-DD HH:mm] [end_time:YYYY-MM-DD HH:mm] [max_members:<n>] [max_accounts_per_user:<n>] [min_townhall:<n>] [max_townhall:<n>] [required-role:<discordRole>] [no-role-signup-limit:<n>] [roster_role:<role>] [allow_multi_signup:<bool>] [sort_by:<signed_up_at|player_name|player_tag|discord_user|townhall>] [import_members:<bool>]` - Create a roster object without posting it yet. CWL/FWA are the supported categories for now and `name` is the preferred command-facing label even though the DB keeps `title`; `title` remains a compatibility alias.
- `/roster list [name:<text>] [user:<discordId>] [player:<playerTagOrName>] [clan:<clanTag>]` - Show roster title, type, clan scope, lifecycle state, posted status, and signup counts for rosters in the guild. When `user` is provided, this becomes a user-scoped signup view that lists that user's linked accounts signed up for current rosters, grouped by roster and roster group. Posted rosters include a clickable link to the original post, and clan values render as `{clan name} (\`{clan tag}\`)` when both are known. This is the public roster-inspection surface; manager mutations stay separate.
- `/roster post roster:<roster>` - Post the signup message for an existing roster, or refresh the existing posted message if it already exists. The posted board uses a compact Player / Discord / Clan table by default plus `Refresh`, `Signup`, `Opt-out`, and `Settings` controls. `Settings` includes `Customize` for saved column-order and sort-mode overrides, `Add User` and `Remove User` roster member panels, and can enable optional `Townhall Icons`, `Index`, `Weight Source`, and `Weight Age` columns.
- `/roster` settings actions currently include export, customize, Change Group, Change Roster, close roster, clear roster, hide buttons, archive mode, unregistered members, and missing members. Destructive actions require confirmation where applicable. Export opens a Google Sheet from the ephemeral roster settings flow.
- `/roster manage roster:<roster> action:<add|move|remove|change_roster|set_weight|open|close|archive> [user:<discordUser>] [group:<groupKey>] [target_group:<groupKey>] [players:<playerTags>]` - Roster-keyed manager controls. `add`, `move`, `remove`, `change_roster`, and `set_weight` open the interactive user/account flow when `user` is provided; `open`, `close`, and `archive` still run directly. `Change Group` (`action:move`) changes selected signups to another group inside the same roster and uses `target_group` as the destination group, defaulting to `Confirmed` when omitted. `Change Roster` (`action:change_roster`) transfers selected signups to another roster. The legacy `group` option still applies to add/ping-style flows, not move. Current weight resolution uses the higher of FWA Stats and manual weight before falling back to lower-confidence roster sources like `PlayerCurrent.currentWeight`.
- `/roster edit roster:<roster> [name:<text>] [category:<CWL|FWA>] [clan:<trackedClanTag>] [timezone:<ianaTz>] [display-timezone:<ianaTz>] [start_time:YYYY-MM-DD HH:mm] [end_time:YYYY-MM-DD HH:mm] [max_members:<n>] [max_accounts_per_user:<n>] [min_townhall:<n>] [max_townhall:<n>] [required-role:<discordRole>] [no-role-signup-limit:<n>] [clear-required-role:<bool>] [roster_role:<role>] [delete_role:<bool>] [allow_multi_signup:<bool>] [sort_by:<signed_up_at|player_name|player_tag|discord_user|townhall>] [import_members:<bool>]` - Edit practical roster metadata, config, and CWL/FWA clan/timezone scope fields. `name` is the supported edit label. `clear-required-role` clears the signup role requirement; `delete_role` only detaches the stored roster-role config and does not delete the Discord role.
- `/roster delete roster:<roster>` - Removes the posted Discord message first when one exists, then hard-deletes a roster and its persisted signup data. If the message cannot be removed, the roster stays intact so you can retry safely.
- `/roster [report|refresh] roster:<roster>` - `report` shows the richer roster status view and `refresh` rerenders the posted board after refreshing rostered players' current-clan state through the service layer.
- `/cwl rotations show [clan:<trackedCwlClanTag>] [day:<1-7>]` - Show active CWL rotation validation across tracked CWL clans, or inspect one clan's planned vs actual lineup one CWL day per page. The no-arg overview includes current-clan leadership names from persisted membership state. The clan autocomplete only lists tracked clans with an active current-season rotation.
- `/cwl rotations create clan:<trackedCwlClanTag> [size:15|30] [roster:<rosterId>] [exclude:<player-tags-separated-by-spaces-or-commas>] [overwrite:true]` - Create or replace the active current-season CWL planner version for one tracked CWL clan during persisted preparation-day state. When `size` is provided it persists the explicit lineup size on the plan. When `roster` is provided, the plan seeds from the confirmed group of one open/closed CWL roster for that clan, but only players present in the clan's current CWL participation data are eligible, and persisted weight data is used when available. Successful creates now show the clan label, source/roster line, included-player count, and excluded-player identities in the response.
- `/cwl rotations delete clan:<trackedCwlClanTag>` - Deactivate the active CWL planner version for one tracked CWL clan in the current season. The clan autocomplete only lists tracked clans with an active current-season rotation.
- `/cwl rotations import sheet:<public-google-sheet-link> [overwrite:true]` - Preview then import active CWL planner tabs from one public Google Sheet into the active-season planner tables. Structural rows may be skipped automatically, but player-like rows are never dropped silently. Import may require a clan-by-clan inline row-review/remap step before save. Public published-sheet imports are read without Google Sheets credentials; export/write still requires writable credentials.
- `/cwl rotations export [new:true|false]` - Export the active CWL planner data to a public Google Sheet using the canonical re-importable tabular format. When omitted or `false`, unchanged same-season exports reuse the previous spreadsheet link instead of creating a new sheet.
- `/accounts [visibility:private|public] [tag:<playerTag>] [discord-id:<@user>]` - List linked player accounts grouped by current clan, with tracked FWA clans shown first. Default is your own account; provide exactly one of `tag` or `discord-id` to inspect a different linked user. `discord-id` uses Discord's native user picker. When inspecting via `tag`, the embed shows `Linked Discord: <@id>` directly under the title. Account names link to the player’s in-game profile, rows use TH badges, show `:crown:` for leaders/co-leaders, and include compact weight from the higher of FWA clan members/catalog values and deferred/manual weight sources, then `PlayerCurrent.currentWeight`, with `—` when no resolved positive weight exists. Pagination keeps each clan group together and uses available embed description space before creating another page.
- `/remindme set type:<WAR|CWL|RAIDS|GAMES> player_tags:<comma-separated-linked-tags> time_left:<HhMm[,HhMm...]> [method:DM|ping-me-here]` - Create durable personal recurring reminder rules for linked player tags only. `time_left` accepts one or more offsets like `12h,2h,30m`; `method` defaults to `DM`. Outbound reminder deliveries are plain-text, and `ping-me-here` includes the inline user mention in the message content so Discord notifies correctly.

## Autorole nickname templates

Autorole can apply Discord nicknames during `/autorole refresh` when nickname sync is enabled:

```text
/autorole config set apply_nicknames:true nickname_template:"{player} | {trackedClans}"
```

Recommended ClashPerk-style template:

```text
{player} | {trackedClans}
```

Example rendered nicknames:

```text
Elrond ♣️ | RR
Elrond ♣️ | RR | GB
Elrond ♣️ | RR | GB | RD | SE | TWC
```

Supported tokens:

| Token | Meaning |
| --- | --- |
| `{player}` | Primary Clash player name. |
| `{tag}` | Primary Clash player tag. |
| `{th}` | Primary account Town Hall. |
| `{clan}` | Primary current clan name. |
| `{clanTag}` | Primary current clan tag. |
| `{clanShort}` | Primary permanent FWA tracked clan short name. |
| `{trackedClans}` | Distinct permanent FWA tracked clan short names across eligible linked accounts, joined with ` | `. |
| `{discord}` | Current Discord server display name. |
| `{username}` | Discord username when available. |
| `{role}` | Primary account in-game clan role. |

Primary account selection prefers eligible linked accounts in permanent FWA tracked clans, then highest Town Hall, then link trust tier, then oldest link, then player tag. `{trackedClans}` de-dupes multiple accounts in the same clan, puts the primary account clan first, then sorts remaining clan short names alphabetically.

Nicknames are cleaned up when tokens are missing and capped to Discord's 32-character nickname limit. Unicode emoji in player names or templates can render in nicknames; custom Discord emoji markup like `<:name:id>` does not render in nicknames and should not be used in templates.

Permissions required for live application:

- The bot needs **Manage Nicknames** to change nicknames.
- The bot's highest role must be above the target member's highest role.
- Role autorole still requires **Manage Roles** and the bot's highest role above every managed role it adds or removes.
